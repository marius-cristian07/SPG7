\section{Frontend Implementation}

The frontend was implemented in the UI layer using the Avalonia UI frameworks together with the MVVM (Model--View--ViewModel) architecture.  
The implementation focused on separating the UI layer from the Domain layer and Data layer.

\subsection{Frontend Architecture}

The frontend was structured into multiple ViewModels and Views: MainViewModel.cs (MainWindow.axaml), AssetViewModel.cs (AssetView.axaml), DashboardViewModel.cs (DashboardView.axaml), ResultViewModel.cs (ResultView.axaml), UnitConfigViewModel.cs and ChartBuilder.cs.



\subsection{Main Navigation and View Handling}

The MainViewModel was used as the center for navigation between different pages of the application. 
The ViewModel has a CurrentViewModel property which determined which page was displayed in the main content area. When the user selects a different tab in the navigation menu the Navigate() method updated the active ViewModel.


\subsection{Asset Management Interface}


The Asset Management page was implemented to allow users to see details of production units, change between scenaios and put units into maintenance.

The page logic was implemented inside the AssetViewModel.  
Each production unit had its own separate UnitConfigViewModel which stored unit specific information such as maintenance and its duration.
Two independent configuration were implemented \_scenario1Configs and \_scenario2Configs.
This was added so that changes made in one scenario did not affect the other scenario.  

 A maintenance check was implemented through the RefreshMaintenanceWarning() method.
Which calculated whether selected maintenance schedules could cause less heat compared to heat demand.
If maintenance risked creating less heat then needed a warning message was displayed to the user.  


\subsection{Data Interface}

The Data page provided users with a overview of system information and operational data. The purpose of this page was to give quick access to key system information before running the optimization.



\subsection{Result Visualization Interface}

The Result page is used to visualize the optimization output.
The page logic was implemented in the ResultViewModel while the graph generation was separated into the ChartBuilder class.

The interface included scenario selection buttons, Winter/Summer selection buttons, graphs for the optimization, graph legends and time labels.

Implemented graphs: heat production, heat demand, CO$_2$ emissions, net production cost and electricity price(For Scenario 2).

The heat production graph used stacked area charts to tell apart production contributions from different units.  
To improve readability individual units were assigned colors and all graphs shared axis configuration methods.  
The graphs dynamically updated whenever the user changed scenarios or season selection.





\subsection{Communication Between Frontend and Backend}

The frontend communicated with backend through the ViewModels. 
The MainViewModel passed the selected scenario,  season, production units and maintenance configurations into the ResultViewModel constructor.

\section{Backend Implementation}

The backend was implemented into the Domain layer of the 3-layer architecture. The backend was responsible for handling optimization logic, maintenance scheduling, production unit management, and processing of operational data used by the UI layer.

Backend consisted of the following classes Optimizer.cs, ProductionUnit.cs, MaintenanceCalculation.cs, MaintenancePeriod.cs, OptimizationResult.cs and ResultFormatter.cs.

\subsection{Backend Architecture}


\subsubsection{Optimizer Class}

The Optimizer class was used for running optimization scenarios and processing data.
Its responsibilities included loading seasonal data, calculating production costs, running optimization scenarios and generating optimization results.

\noindent \newline The optimizer received data from the Data layer through a dependency injection:

\begin{verbatim}
public Optimizer(ISourceDataManager sourceDataManager,
                 IAssetManager assetManager)
\end{verbatim}
The optimizer maintained separate dictionaries for Summer and Winter.



\subsubsection{ProductionUnit Class}

The ProductionUnit class implemented individual heat production units.
Each production unit contained maximum heat production capacity, electricity generation, production costs, energy consumption and CO$_2$ emissions.

\noindent \newline Availability check was implemented:

\begin{verbatim}
public bool IsAvailable(DateTime time)
\end{verbatim}
This method made sure that units under maintenance were excluded form the optimization.

\subsubsection{MaintenanceCalculation Class}

\hbox{The MaintenanceCalculation class was used for handling maintenance for production units.}

The class created maintenance periods based on the users choices.
MaintenanceCalculation class located the selected production unit then created a maintenance period and assigned the maintenance period to the unit.


\subsubsection{ResultFormatter Class}

The ResultFormatter class was used for generating summerized optimization results.
The implementation formatted optimization results into summaries that included the selected optimization scenario, seasonal configuration, maintenance schedules and hourly production schedules.


\subsection{Business Logic Implementation}

Two optimization scenarios were implemented.

\subsubsection{Scenario 1 -- Production Cost Optimization}

Scenario 1 optimized heat production by production cost.
The optimizer first checked production units for maintenance then it sorted the units by production cost after which it allocated heat production sequentially and continued until heat demand was satisfied.

\subsubsection{Scenario 2 -- Net Production Cost Optimization}

Scenario 2 extended the optimization logic by including electricity prices.

\noindent \newline The net production costs are calculated with this formula:

\begin{equation}
\mathrm{Net\ Production\ Cost} =
\mathrm{Production\ Cost} -
\left(
\frac{\mathrm{Electricity\ MW}}
{\mathrm{Maximum\ Heat\ MW}}
\times
\mathrm{Electricity\ Price}
\right)
\end{equation}
This allowed the units that generated electricity to be prioritized. Unit rankings were updated every hour using the current electricity prices.

\subsection{Exception handling}

Exception handling was implemented to make sure the optimization can run.
Implemented exceptions include checking if the production unit had a MaxHeatMW and whether or not source data exist for a given time.


\noindent \newline If the production unit did not have MaxHeatMW an exception was thrown:

\begin{verbatim}
if (unit.MaxHeatMW <= 0)
{
    throw new ArgumentException(...);
}
\end{verbatim}


\subsection{Communication with the Data Layer}

The backend communicated with the Data layer through interfaces ISourceDataManager and IAssetManager.
They provided access to seasonal dictionaries and production unit information.
The loading of data was separated into the Data layer.

\section{Data Management}

The data management was implemented into the Data layer of the 3-layer architecture. It was used for managing operational data, production unit configurations and communication between the backend and external data.

The Data layer was split into three primary components the Asset Manager, Source Data Manager and Result Data Manager.

\subsection{Data Layer Structure}

The Data layer consisted of the following classes and interfaces AssetManager.cs, IAssetManager.cs, SourceDataManager.cs, ISourceDataManager.cs, ResultDataManager.cs, IResultDataManager.cs and DataPoint.cs.
This layer also included external CSV files SummerSourceDataSheet.csv and WinterSourceDataSheet.csv.


\subsection{Asset Manager}

The AssetManager class was used for implementation of production unit information.
It also managed a list of ProductionUnit objects.
Each production unit contained a unit name, maximum heat production, electricity production, energy consumption, production cost, CO$_2$ emissions and image path.

\noindent \newline An example of production unit implementation:

\begin{verbatim}
new ProductionUnit
{
    Name = "GB1",
    ImagePath = $"{imagePath}/unit1.png",
    MaxHeatMW = 3.0,
    ElectricityMW = 0,
    EnergyConsumption = 1.05,
    ProductionCost = 510,
    CO2Emissions = 132
}
\end{verbatim}
The AssetManager provided access to all production units through:

\begin{verbatim}
public List<ProductionUnit> GetProductionUnits()
\end{verbatim}


\subsection{Source Data Manager}

The Source Data Manager was used for loading and processing of data from CSV files.
The class loaded two dictionaries (Winter and Summer) from the CSV files. Dictionaries were implemented as "Dictionary<DateTime, DataPoint> Winter/Summer" where each timestamp was attached to a DataPoint containing heat demand and electricity price.



\subsection{Result Data Manager}

The Result Data Manager class was used for storing optimization results. 
Result Data Manager stored the scenario number, seasonal configuration, time, production unit name, heat production and CO2 emissions for each hour in the optimization.



\subsection{Exception handling}

Exception handling was implemented to make sure data can be read.
Implemented exception includes verifying if the CSV file exists.


\noindent \newline If the file could not be found an exception was thrown:

\begin{verbatim}
if (!File.Exists(fileName))
    throw new FileNotFoundException(...);
\end{verbatim}


\subsection{Communication with the Backend}

The Data layer communicated with the backend through a dependency injection. This allows the backend to access seasonal dictionaries, production units and structured optimization results.
