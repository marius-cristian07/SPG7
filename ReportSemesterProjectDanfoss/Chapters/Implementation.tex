\section{Frontend Implementation}


The frontend was implemented into the UI layer using the Avalonia UI framework~\cite{avalonia} together with the MVVM (Model--View--ViewModel)~\cite{avaloniamvvm}~\cite{mvvmtoolkit} architecture. The purpose of the frontend was to provide operators and users with an intuitive interface for selecting optimization scenarios, and visualizing optimization results in a structured and user-friendly way.

The implementation focused on separating the UI layer from the Domain layer and Data layer in order to improve scalability, and code readability. 

\subsection{Frontend Architecture}

The frontend was structured into multiple ViewModels and Views:

\begin{itemize}
    \item MainViewModel.cs (MainWindow.axaml)
    \item AssetViewModel.cs (AssetView.axaml)
    \item DashboardViewModel.cs (DashboardView.axaml)
    \item ResultViewModel.cs (ResultView.axaml)
\end{itemize}
Each ViewModel was also connected to the corresponding Avalonia XAML view.
This separation ensured that UI components handled presentation only, while the ViewModels managed the current screen, navigation, and interaction logic.

\subsection{Main Navigation and View Handling}

The MainViewModel served as the central controller for navigation between different pages of the application. 

The ViewModel maintained a CurrentViewModel property, which determined which page was displayed in the main content area. When the user selected a different tab in the navigation menu, the Navigate() method updated the active ViewModel accordingly.

The tabs were stored using the currently selected tab through the SelectedTab property. 

\noindent \newline  The boolean properties:

\begin{itemize}
    \item IsAssetTabSelected
    \item IsDashboardTabSelected
    \item IsResultTabSelected
\end{itemize}
were used to provide visual feedback

\subsection{Asset Management Interface}

The Asset Management page was implemented to allow users to:

\begin{itemize}
    \item See details of production units
    \item Change between scenaios 
    \item Put units into maintenance
\end{itemize}
The AssetViewModel was used to put units into maintenance which was later used by the Domain layer to remove them form the optimization for the given time.

\subsection{Data Interface}

The Data page provided users with a high-level overview of system information and operational data. The purpose of this page was to give quick access to key system metrics before running optimization.

The dashboard implementation focused on clarity and readability through showing only key information about units


\subsection{Result Visualization Interface}

The Result page is used to visualize the optimization output generated by the backend.

\noindent \newline The implementation included interactive graphs for:

\begin{itemize}
    \item Heat production
    \item Heat demand
    \item CO2 emissions
    \item Net production cost
    \item Electricity price (For scenario 2)
\end{itemize}
The graphs were designed using the charting libraries selected for the project~\cite{livecharts}~\cite{skiasharp} to help users quickly identify production patterns and compare optimization results across different scenarios.

\noindent \newline The interface included:

\begin{itemize}
    \item Scenario selection buttons
    \item Winter/Summer selection buttons
    \item Dynamic graphs
    \item Legends for graphs
\end{itemize}
The graph visualization clearly distinguished between different heat production units using colors and layered chart areas.

\subsection{Communication Between Frontend and Backend}

The frontend communicated with backend through the ViewModels. The UI itself did not directly perform optimization calculations instead it sent information user selected to the backend.

\noindent \newline When the user navigated to the Result page, the MainViewModel collected:

\begin{itemize}
    \item Selected scenario
    \item Selected season
    \item Selected units (Units of the selected scenario)
\end{itemize}

\noindent These values were then passed into the ResultViewModel constructor:

\begin{verbatim}
CurrentViewModel = new ResultViewModel(
    AssetPage,
    scenario,
    isSummer,
    selectedUnits
);
\end{verbatim}
This implementation ensured loose coupling between UI components and the backend.

\section{Backend Implementation}

The backend was implemented into the Domain layer of the 3-layer architecture. The backend was responsible for handling optimization logic, maintenance scheduling, production unit management, and processing operational data used by the frontend UI layer.

\noindent \newline The backend structure consisted of the following classes:

\begin{itemize}
    \item Optimizer.cs
    \item ProductionUnit.cs
    \item MaintenanceCalculation.cs
    \item MaintenancePeriod.cs
    \item ResultDM.cs
\end{itemize}

\subsection{Backend Architecture and Responsibilities}

The backend followed a layered structure where each class had a dedicated responsibility.

\subsubsection{Optimizer Class}

The Optimizer class was responsible for running optimization scenarios and processing operational data.

\noindent \newline Its primary responsibilities included:

\begin{itemize}
    \item Loading seasonal operational data
    \item Managing production units
    \item Calculating production costs
    \item Running optimization scenarios
    \item Generating scheduling results
\end{itemize}
The optimizer received operational data from the Data layer through dependency injection:

\begin{verbatim}
public Optimizer(ISourceDataManager sourceDataManager,
                 IAssetManager assetManager)
\end{verbatim}
\hbox{This implementation reduced coupling between system components and improved maintainability.}
The optimizer maintained separate dictionaries for seasonal datasets:

\begin{itemize}
    \item Summer
    \item Winter
\end{itemize}
Each dataset contained hourly operational values:

\begin{itemize}
    \item Heat demand
    \item Electricity prices
\end{itemize}

\subsubsection{ProductionUnit Class}

The ProductionUnit class represented individual heat production units within the system.

\noindent \newline Each production unit contained:

\begin{itemize}
    \item Maximum heat production capacity
    \item Electricity generation
    \item Production costs
    \item Energy consumption
    \item CO$_2$ emissions
\end{itemize}
Availability checks were implemented using the IsAvailable() method:

\begin{verbatim}
public bool IsAvailable(DateTime time)
\end{verbatim}
This method ensured that units under maintenance were excluded form the optimization.

\subsubsection{MaintenanceCalculation Class}

\hbox{The MaintenanceCalculation class handled maintenance scheduling for production units.}

The class created maintenance periods dynamically based on user-defined parameters from the frontend interface.

\noindent \newline The scheduling logic:

\begin{itemize}
    \item Located the selected production unit
    \item Created a maintenance period
    \item Assigned the maintenance interval to the unit
\end{itemize}


\subsubsection{ResultDM Class}

The ResultDM class functioned as a data model for transferring optimization results between backend components and the frontend UI layer.

\noindent \newline The class stored:

\begin{itemize}
    \item Production unit information
    \item Net production cost
    \item Heat production values
\end{itemize}


\subsection{Business Logic Implementation}

The backend implemented two primary optimization scenarios.

\subsubsection{Scenario 1 -- Production Cost Optimization}

Scenario 1 optimized heat production by production cost.

\noindent \newline The algorithm:

\begin{enumerate}
    \item Retrieved current heat demand
    \item Filtered available production units
    \item Sorted units by production cost
    \item Allocated heat production sequentially
    \item Continued until demand was satisfied
\end{enumerate}

\subsubsection{Scenario 2 -- Net Production Cost Optimization}

Scenario 2 extended the optimization logic by including electricity prices into the calculation.

\noindent \newline The net production costs are calculated using:

\begin{equation}
\mathrm{Net\ Production\ Cost} =
\mathrm{Production\ Cost} -
\left(
\frac{\mathrm{Electricity\ MW}}
{\mathrm{Maximum\ Heat\ MW}}
\times
\mathrm{Electricity\ Price}
\right)
\end{equation}
This allowed the system to prioritize units capable of generating value through electricity production.
The implementation dynamically recalculated unit ranking for every hour based on current electricity prices.

\subsection{Data Processing and Validation}

The backend performed several validation and processing operations to ensure stable system behavior.

\noindent \newline Examples included:

\begin{itemize}
    \item Validation of production unit capacity
    \item Verification of available source data
    \item Maintenance availability checking
    \item Demand allocation calculations
\end{itemize}
For example, the optimizer validated heat production capacity before performing calculations:

\begin{verbatim}
if (unit.MaxHeatMW <= 0)
{
    throw new ArgumentException(...);
}
\end{verbatim}
Similarly, the system verified that operational data existed for the requested time before running the optimization.

\subsection{Communication with the Data Layer}

The backend communicated with the Data layer through interfaces:

\begin{itemize}
    \item ISourceDataManager
    \item IAssetManager
\end{itemize}
These interfaces provided access to:

\begin{itemize}
    \item Seasonal operational datasets
    \item Production unit configurations
    \item Asset information
\end{itemize}
The actual loading of data was separated into the Data layer. 
This separation improved modularity and allowed backend optimization logic to remain independent from data layer.

\section{Data Management}

The data management was implemented into the Data layer of the 3-layer architecture. It was responsible for managing operational datasets, production unit configurations, and structured communication between the backend and external data.


\noindent \newline The Data layer was divided into two primary components:

\begin{itemize}
    \item Asset Management
    \item Source Data Management
\end{itemize}


\subsection{Data Layer Structure}

The Data layer consisted of the following main classes and interfaces:

\begin{itemize}
    \item AssetManager.cs
    \item IAssetManager.cs
    \item SourceDataManager.cs
    \item ISourceDataManager.cs
    \item DataPoint.cs
\end{itemize}
The layer also included external CSV files used as operational datasets:

\begin{itemize}
    \item SummerSourceDataSheet.csv
    \item WinterSourceDataSheet.csv
\end{itemize}
This structure separated static asset information from time-series operational data.

\subsection{Asset Manager}

The Asset Manager component handled storage and retrieval of production unit information used by the optimizer.

The AssetManager class implemented the IAssetManager interface and maintained a collection of ProductionUnit objects.

\noindent \newline Each production unit contained operational parameters such as:

\begin{itemize}
    \item Unit name
    \item Maximum heat production
    \item Electricity production
    \item Energy consumption
    \item Production cost
    \item CO$_2$ emissions
    \item Image path for frontend visualization
\end{itemize}
An example production unit definition was implemented as:

\begin{verbatim}
new ProductionUnit
{
    Name = "GB1",
    MaxHeatMW = 3.0,
    ProductionCost = 510,
    CO2Emissions = 132
}
\end{verbatim}
The AssetManager provided access to all production uinits through:

\begin{verbatim}
public List<ProductionUnit> GetProductionUnits()
\end{verbatim}


\subsection{Source Data Manager}

The Source Data Manager handled loading and processing of operational datasets from CSV files.

\noindent \newline The SourceDataManager class implemented the ISourceDataManager interface and loaded two datasets:

\begin{itemize}
    \item Summer operational data
    \item Winter operational data
\end{itemize}
The datasets were stored internally as dictionaries:

\begin{verbatim}
Dictionary<DateTime, DataPoint>
\end{verbatim}
Each timestamp mapped to a DataPoint object containing:

\begin{itemize}
    \item Heat demand
    \item Electricity price
\end{itemize}
The DataPoint model was implemented as:

\begin{verbatim}
public class DataPoint
{
    public double HeatDemand { get; set; }
    public double ElectricityPrice { get; set; }
}
\end{verbatim}
This structure enabled efficient lookup of operational values during optimization.

\subsection{CSV Data Loading}

Operational datasets were loaded from CSV files during application startup.

\noindent \newline The backend resolved file locations dynamically using:

\begin{verbatim}
ResolveSourceDataPath()
\end{verbatim}

\hbox{This implementation improved portability by supporting multiple deployment paths.}

\noindent \newline The CSV loading process included:

\begin{enumerate}
    \item Opening the selected file
    \item Skipping the header row
    \item Reading each line sequentially
    \item Parsing timestamps and numeric values
    \item Creating DataPoint objects
    \item Storing the results in dictionaries
\end{enumerate}
The parsing logic converted the CSV values into structured data objects:

\begin{verbatim}
DateTime time = DateTime.Parse(parts[0]);
double heat = double.Parse(parts[1]);
double price = double.Parse(parts[2]);
\end{verbatim}
Which were later used in the optimization.

\subsection{Data Validation and Consistency}

Several validation checks were implemented to ensure stable and consistent data handling.

\noindent \newline Examples included:

\begin{itemize}
    \item Verification that CSV files existed before loading
    \item Exception handling for invalid numeric values
\end{itemize}
If a dataset file could not be found, the system generated an exception:

\begin{verbatim}
throw new FileNotFoundException(...)
\end{verbatim}
This approach prevented the program from completly crashing if the file is not found.

\subsection{Integration with the Backend}

The Data layer supplied operational datasets and production unit information to the backend.

\noindent \newline This allowed the optimizer to use:

\begin{itemize}
    \item Seasonal operational datasets
    \item Production unit configurations
\end{itemize}
during the optimization.
