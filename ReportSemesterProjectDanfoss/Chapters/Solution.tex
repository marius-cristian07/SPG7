In this chapter we will present important UML diagrams to document the design of our application.
Before we started coding, we started working on diagrams for our heating optimization application.  Our system should minimize operational expenses and satisfy the heat demand while considering electricity market and maintenance constraints. The diagrams help to structure the systems and make critical architectural design decisions. 
Based on the requirements, we structure our system in a modular architecture with an Asset Manager, Source Data Manager, Optimizer, Result Data Manager, and Data Visualization module to isolate the responsibilities into dedicated components. 
The use of our MVVM pattern separates UI and business logic. Furthermore, by applying the Single Responsibility Principle, we can ensure maintainability. 
From different points of view, the architecture is shown by the package diagram, the workflow by the activity diagram, the interactions between objects and system's components by the sequence diagram, and the internal object-oriented design and structure by the class diagram. 
% --------------------------------
\section{Component Diagram}

The component diagram shows the architectural structure of the system. Given the requirements, we need to have an Asset Manager, Source Data Manager, Optimizer, Result Data Manager, and Data Visualization. 
The asset manager handles static information such as the production unit, production cost, and CO$_2$ emissions. 
The Source Data Manager contains dynamic time series data such as heat demand and electricity prices. 
The separation of AM and SDM is split up in this way to make sure that it reduces coupling between configuration data and operational input data. Furthermore, both sources of static and dynamic data can be expanded independently. 
The optimizer component contains the logic of optimization. This approach centralizes the business logic in a single component, enabling modularity. The implementation also allows testing to be done separately from the user interface. The data input/output of the optimization is reduced to available production units, heat demand, electricity prices, and maintenance constraints, without knowing about charts or navigation in the UI. 
Both optimization scenarios are implemented as encapsulated solutions in accordance with Danfoss’s specifications. Scenario 1 prioritizes the production cost, and scenario 2 prioritizes the net production cost. 
The Data Visualization component handles data for visualization, such as graphs. The key advantage of this design is changing the view, such as Chart Framework, without affecting the optimization logic. 
The system operates by separating configuration data, operational input data, and optimization results into repositories to improve maintainability and data management. 
This architecture supports future decisions such as adding new production units, creating new optimization strategies, and adding new visualizations.

\section{Activity Diagram}

This chosen activity diagram highlights the entire process, from user configuration to the display of results. To put it more accurately, it models the UI workflow combined with the optimization process within the MVVM architecture. 
This workflow shows the central use case of the system and connects scenario selection, maintenance, the optimizer, and the result view tab. 
The workflow starts with selecting the scenario and the season and configuring the maintenance period. 
The Asset View Model validates the maintenance configuration and checks if the heat demand can be satisfied.
When the user opens the result tab the ResultViewModel prepares the optimization parameters and applies the maintenance constraints to the production units. 
The optimizer checks which scenario is selected and executes the correct workflow. 
When scenario 1 is chosen, the optimizer prioritizes units based on production cost and generates a heat-only production schedule, as it performs a heat-only optimization without electricity generation or consumption. 
In scenario 2, the optimizer performs a heat and electricity optimization by prioritizing production units based on net production cost, which depends on electricity prices and unit type, and generates a production schedule for heat and electricity operation.
After the generation of the results, the ResultViewModel creates the report and prepares the visualization data for presentation. 
The user can view the optimization results and charts. 
Overall, the activity diagram document the complete optimization workflow and highlights the relationship between maintenance, optimization, and result generation.

\section{Sequence Diagram}


The sequence diagram documents the communication and interaction between objects in general. The selected sequence diagram models the same use case as the activity diagram, from the user interaction in the UI to the optimization and visualization of the results. In more detail, it shows how the ViewModels coordinate the workflow, how the optimizer calculates the schedule, and how the result data is transformed into text and charts. The key advantage of this sequence diagram is that it shows the separation of concerns and how components communicate within the MVVM pattern. 
The diagram contains the following participants: User, MainViewModel, AssetViewModel, ResultViewModel, Optimizer, ProductionUnit, ResultFormatter, and ChartBuilder. 
The user starts the interaction by selecting the result tab, which triggers the MainViewModel navigation command. Then the MainViewModel reads the selected scenario from the AssetViewModel and requests the selected unit names from the current scenario. After that, it creates a new ResultViewModel with the selected scenario, the season, and the selected units as parameters. 
If the AssetViewModel is available in the ResultViewModel, it prepares the optimization by applying the maintenance configuration. The ResultViewModel then calls the optimizer with the selected scenario and allowed unit names. The optimizer calculates and checks each hour each ProductionUnit for availability and maintenance constraints inside the optimization loop. 
An alt section divides scenario 1 from scenario 2 because they use different prioritization strategies. For scenario 1, the optimizer runs a heat-only optimization and prioritizes the units by production cost. For scenario 2, the optimizer runs a heat and electricity optimization and prioritizes the units by net production cost, which depends on electricity prices and unit type. 
After the optimization result is generated, the ResultViewModel forwards the data to the ResultFormatter for the textual report and to the ChartBuilder for the chart data. The ResultFormatter creates the report text, and the ChartBuilder creates the series for the visualizations. 
Overall, the sequence diagram shows how the responsibilities are separated between navigation, configuration, optimization, formatting, and visualization. Apart from the activity diagram, it focuses on communication between system objects and the distribution of responsibilities.

\section{Class diagram}
The class diagram represents a static structure of our system. Especially this one shows the central classes, interfaces, attributes, methods, and relationships between classes. This version is simplified for readability. The version that implemented all classes is in the appendix.  The diagram models the architecture of our heat production optimization application. We included all the components (AM, SDM, OPT, DV, RDM), which are explained in more detail at the beginning of this chapter.  
The asset manager component is responsible for providing the repository of production units and their static technical properties.
The asset manager contains the following classes:

    The IAssetManager is an interface of the asset manager class, which defines access to production unit data. The reason why we used an interface for the assetmanager is, to allow diffrent asset sources
    The AssetManager class is an implemented class from the IAssetManager interface. It acts as a repository for the production units. 
    The ProductionUnit class is one of the domain classes. It represents a production unit and contains the following information: heat production, electricity production, production cost, CO$_2$ emissions, and maintenance periods. The production unit can be a gas, oil, or electric boiler or a gas motor. 
    The MaintenancePeriod class displays scheduled maintenance periods while defining start and end times. It affects the availability of the production units.

The relationship between AssetManager and ProductionUnit is a one-to-many relationship because the asset manager provides many production units. 
The ProductionUnit has a composition relationship with MaintenancePeriod. A ProductionUnit may have zero or more MaintenancePeriods and each MantenancePeriod belongs to one ProductionUnit.

The source data manager manages dynamic timeseries data. 
The source data manager contains the following classes:

    The ISourceDataManager interface defines the access to the time series data. This encapsulation enables the use of other sources in the future like api. 
    The SourceDataManager class loads the scenario data and manages the dataset of winter and summer periods. 
    The DataPoint class represents the input data for one timestamp, which contains the heat demand and electricity price. It is the foundation of the hourly optimization calculation. 

The SourceDataManager has a one-to-many relationship with DataPoint because for every winter and summer data set, there are many hourly data points. 

The optimizer contains the business logic of the system and makes, on an hourly basis, scheduling decisions that consider heat demand, electricity prices, maintenance, and unit availability. 
The optimizer contains the following classes:

    The optimizer class implements both scenarios separately. The step-by-step process of both optimizations is explained in more detail in the activity diagram.  Furthermore, it calculates the net production cost and prioritizes production units. The net production cost is fundamental for the optimizer logic. It combines production cost with electricity cost and depends dynamically on electricity prices. The optimization is done on an hourly basis. 
    The OptimizationResult class is a small helper model (unit, net cost, heat) used for ranking by net production cost. Note: the scenario methods (RunScenario1, RunScenario2) return schedules as tuples (hour + per-unit allocations) which are passed to visualization and export; OptimizationResult` is not the primary output container for those scenario runs.

The optimizer depends on ProductionUnit and DataPoint to run the hourly scheduling and produces per-hour schedules (tuples) consumed by the UI and export logic.

The data visualization component implements UI logic of the system and is based on the MVVM architecture. 
The data visualization contains the following classes:

    The MainViewModel class is responsible for the central navigation and manages the tabs. 
    The AssetViewModel class handles user configuration such as selected scenario and maintenance setup. Furthermore, it prepares optimization parameters. 
    The ResultViewModel class processes the results of the optimization and manages scenario switching, season switching, and report loading. Overall, it connects the optimizer with the visual layer. 
    The ChartBuilder class is responsible for building the charts (heat production charts, CO$_2$ charts, electricity charts, and net cost charts).
    The ResultFormatter class is responsible for the text report, such as creating reports independently from the optimizer. 

The ResultViewModel has a relationship with the optimizer because it is dependent for calculation results on ChartBuilder for chart data and on ResultFormatter for the textual report. 

The result data manager component saves the optimized results. 
The IResultDataManager interface defines the export API and ResultDataManager provides a concrete CSV export implementation.

Overall, the class diagram shows our final static architecture and how responsibilities are separated between data management, optimization, and visualization.

The application was built using the Model-View-ViewModel (MVVM) architectural pattern because it provides a strict separation of concerns for organizing our code and makes it easy to separate the user interface from the actual code logic since our project involved both a visual dashboard and an optimization system, which helped keep the codebase manageable as it grew.

\subsection{The Technology Stack}
To build the application we used C\# making use of the object-oriented structure to represent the different heat production units and optimization rules in our system.

For the user interface we chose Avalonia UI because it fits the MVVM design pattern perfectly and to create the graphs we used the LiveCharts2 library since a large part of the project involved presenting optimization results in a way that was easy for users to understand.

For data storage since the project was based on two static CSV files, WinterSourceDataSheet.csv and SummerSourceDataSheet.csv, our application read them directly into memory and because the data was static and did not need to be changed by users, a database would have added extra complexity without any benefits.

\subsection{Separation of Concerns}

To make the project easier to understand and work on as a team we devided the app into four layers: the View Layer, the ViewModel Layer, the Domain Layer, and the Data Access Layer

Only the appearance of the program is controlled by the view layer as it contains files that specify UI elements including grids, buttons, sliders, date pickers, and more, like AssetView.axaml and ResultView.axaml.

Data binding connects the UI to the ViewModels, like when a user clicks a button, the ViewModel collects the input data, sends it to the backend logic and after receiving the results it updates the UI automatically.

The domain layer doesn't know about the user interface and focuses just on calculations, in short being the logic of the app.

The Data Layer is loading the data into the system by reading the CSV files and converting them into usable structures like lists.

\subsection{Rationale for the Chosen Structure}

Using this separated structure helped us reduce the dependencies between parts of the code and it also helped with working on different parts of the app at the same time while not running into problems and merge conflicts.

For the data, again instead of introducing a database we simply loaded our fixed CSV files as lists when the application starts and it made things much more simple for the current scope of this project since there was no real need for a more complex solution.
